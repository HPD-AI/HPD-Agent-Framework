# HPD-Agent Framework Changelog: v0.5.0 -> v0.5.5

## Overview

This release continues HPD-Agent's pre-1.0 API evolution. Until `1.0.0`, the framework may refine public APIs, persistence shapes, hosting contracts, and UI extension points as the runtime model settles.

The primary theme in `v0.5.5` is the current conversation architecture: **sessions contain threads**, and **thread event streams are the durable source of conversation history**. The older branch terminology has been formalized into thread APIs across core, hosting, ASP.NET, TUI, audio projection, evaluations, bots, and sub-agents.

Major themes:

- **Threads as the conversation model**: Branch APIs and route names are now thread APIs.
- **Event-sourced thread persistence**: Session metadata is stored separately from thread event streams.
- **Runtime and hydration hardening**: Session hydration, streaming, and active-run tracking were tightened.
- **TUI v2 composition**: Transcript rendering now uses cells and renderer registrations.
- **Provider setup in TUI**: Model/provider selection now has a catalog-driven surface.
- **Pre-1.0 compatibility posture**: These changes are expected before `1.0.0` and should be documented as the current API model.

---

## Major Changes

### 1. Session Branches Are Now Session Threads

**Status**: PRE-1.0 API EVOLUTION

The framework now uses thread terminology for conversation paths. A `Session` owns one or more `Thread` objects, and each thread represents a durable conversation path.

Common rename pattern:

```text
Branch -> Thread
branches -> threads
BranchDto -> ThreadDto
IAgentBranchService -> IAgentThreadService
AgentBranchService -> AgentThreadService
BranchRunDto -> ThreadRunDto
BranchRunStartedEvent -> ThreadRunStartedEvent
BranchRunCompletedEvent -> ThreadRunCompletedEvent
BranchHistoryCompaction -> ThreadHistoryCompaction
StateScope.Branch -> StateScope.Thread
BeforeBranchForkCommitAsync -> BeforeThreadForkCommitAsync
```

The current model should be documented as:

```text
Session
  Thread: main
  Thread: edited/forked path
  Thread: sub-agent/runtime child path
```

#### Migration

Update code that references branch types, branch services, or branch IDs to the corresponding thread APIs.

```csharp
// Before
var branches = await branchService.ListBranchesAsync(sessionId);

// After
var threads = await threadService.ListThreadsAsync(sessionId);
```

---

### 2. ASP.NET Routes Use `/threads`

**Status**: PRE-1.0 API EVOLUTION

Hosted API routes now expose thread endpoints.

Current route shape:

```text
GET    /sessions/{sid}/threads
GET    /sessions/{sid}/thread-graph
GET    /sessions/{sid}/threads/{threadId}
POST   /agents/{agentId}/sessions/{sid}/threads
POST   /agents/{agentId}/sessions/{sid}/threads/{threadId}/fork
PATCH  /sessions/{sid}/threads/{threadId}
DELETE /sessions/{sid}/threads/{threadId}
GET    /sessions/{sid}/threads/{threadId}/events
```

Content routes are also thread-scoped:

```text
POST   /sessions/{sid}/threads/{threadId}/content
GET    /sessions/{sid}/threads/{threadId}/content
GET    /sessions/{sid}/threads/{threadId}/content/{contentId}
DELETE /sessions/{sid}/threads/{threadId}/content/{contentId}
```

#### Migration

Clients using `/branches` routes should move to the `/threads` equivalents. Endpoint names also use `Thread` naming, such as `ListThreads`, `GetThread`, `ForkThread`, and `GetThreadEvents`.

---

### 3. Session Persistence Is Event-Sourced by Thread

**Status**: PRE-1.0 API EVOLUTION

`ISessionStore` now separates session metadata from thread history.

Current persistence model:

- `Session` stores session-level metadata and session-scoped middleware state.
- `Thread` stores projected messages, thread metadata, and thread-scoped middleware state.
- `ThreadEventDocument` stores the durable event stream for a thread.
- Thread runs and recovery state are projected from thread events.

Current key APIs:

```csharp
Task<Session?> LoadSessionAsync(string sessionId, CancellationToken cancellationToken = default);
Task SaveSessionAsync(Session session, CancellationToken cancellationToken = default);

Task<Thread?> LoadThreadAsync(string sessionId, string threadId, CancellationToken cancellationToken = default);
Task<ThreadEventDocument?> LoadThreadDocumentAsync(string sessionId, string threadId, CancellationToken cancellationToken = default);
Task AppendThreadEventAsync(string sessionId, string threadId, AgentEvent evt, long? expectedSequenceNumber = null, CancellationToken cancellationToken = default);
IAsyncEnumerable<AgentEvent> ReadThreadEventsAsync(string sessionId, string threadId, ReplayReadOptions options, CancellationToken cancellationToken = default);
Task<List<string>> ListThreadIdsAsync(string sessionId, CancellationToken cancellationToken = default);
Task DeleteThreadAsync(string sessionId, string threadId, CancellationToken cancellationToken = default);
```

#### Removed

`UncommittedTurn` has been removed. In-flight and recovery state now comes from thread run events and projections.

#### Migration

Custom `ISessionStore` implementations must implement thread document/event methods and stop relying on session-level message history or uncommitted turn files.

### 4. Thread Fork Graph

**Status**: CURRENT MODEL

Thread forks are represented as semantic fork groups rather than direct parent-only navigation.

Use:

```csharp
ThreadForkGraph.BuildVisibleForkGroups(...)
```

Use this when rendering alternate conversation choices in a TUI, web app, or desktop host. `Thread.ForkedFrom` remains direct lineage; fork groups answer where a user should see alternate choices in the transcript.

---

### 5. Middleware Hook and State Scope Naming

**Status**: PRE-1.0 API EVOLUTION

Middleware lifecycle hooks now use thread terminology.

```csharp
// Before
Task BeforeBranchForkCommitAsync(
    BeforeBranchForkCommitContext context,
    CancellationToken cancellationToken);

// After
Task BeforeThreadForkCommitAsync(
    BeforeThreadForkCommitContext context,
    CancellationToken cancellationToken);
```

Persistent state scope naming also changed:

```csharp
// Before
StateScope.Branch

// After
StateScope.Thread
```

`StateScope.Thread` is the default conversation-path scope. `StateScope.Session` is still for state shared across all threads in a session.

### 6. TUI v2 Transcript Composition

**Status**: API EVOLUTION

The TUI package now uses transcript cells and renderer registrations for customizable output.

New concepts include:

- `TranscriptCell`
- `TranscriptEntry`
- `AgentTuiTranscriptRendererRegistry`
- `IAgentTuiTranscriptRenderer<TCell>`
- `AgentTuiTranscriptRenderContext<TCell>`
- `AgentTuiTranscriptRenderServices`

This enables package-specific transcript cells without hard-coding every display type into the core transcript view.

#### Coding TUI Updates

Coding harness TUI output now has dedicated cells/renderers for command execution, exploration, diagnostics, and file mutations.

---

### 7. TUI Model and Provider Selection

**Status**: NEW SURFACE

The TUI package now includes a catalog-driven model selection surface.

Key types:

```csharp
public interface IAgentTuiModelCatalog
{
    ValueTask<IReadOnlyList<AgentTuiProviderChoice>> GetProvidersAsync(...);
    ValueTask<IReadOnlyList<AgentTuiModelChoice>> GetModelsAsync(...);
}
```

Related types:

- `AgentTuiProviderChoice`
- `AgentTuiModelChoice`
- `AgentTuiModelQuery`
- `AgentTuiSelectedModel`
- `AgentTuiModelSelectionState`

`HPD-Agent.ModelsDev` no longer owns a TUI-specific catalog implementation. Applications can adapt `ModelsDevStore` or another source into `IAgentTuiModelCatalog`.

---

### 8. Provider Configuration Uses ProviderOptions

**Status**: API EVOLUTION

Provider-specific configuration is now represented through `ProviderOptions` and source-generated provider config helpers.

```csharp
public JsonElement? ProviderOptions { get; set; }

public T? GetProviderConfig<T>() where T : class;
public void SetProviderConfig<T>(T config) where T : class;
```

`ClientProviderConfig.AdditionalProperties` was removed. Use typed provider config objects, `ProviderOptions`, or runtime `ChatOptions.AdditionalProperties` depending on whether the setting belongs to stored agent config or a single model call.

---

### 9. Sub-Agent Execution Policies Use Threads

**Status**: API EVOLUTION

Sub-agent execution policy names now align with session/thread terminology.

Current concepts:

```csharp
SubAgentSessionPolicy.ParentSession
SubAgentSessionPolicy.NewSession
SubAgentSessionPolicy.SharedSession

SubAgentThreadPolicy.ForkFromParentThread
SubAgentThreadPolicy.FreshThread
SubAgentThreadPolicy.ExistingThread
SubAgentThreadPolicy.ParentThread
```

Helper methods include:

```csharp
SubAgentExecutionPolicies.ParentSessionForkedThread()
SubAgentExecutionPolicies.ParentSessionFreshThread()
SubAgentExecutionPolicies.ParentThread()
SubAgentExecutionPolicies.SharedSessionFreshThread(sessionId)
SubAgentExecutionPolicies.SharedSessionExistingThread(sessionId, threadId)
SubAgentExecutionPolicies.ExistingThread(threadId)
```

## Migration Checklist

Use this checklist when moving from `v0.5.0` to `v0.5.5`:

- Replace branch type names with thread type names.
- Replace `/branches` API calls with `/threads` API calls.
- Replace branch run DTOs/events with thread run DTOs/events.
- Update custom `ISessionStore` implementations for thread event documents.
- Remove `UncommittedTurn` dependencies.
- Rename `BeforeBranchForkCommitAsync` middleware hooks to `BeforeThreadForkCommitAsync`.
- Replace `StateScope.Branch` with `StateScope.Thread`.
- Update sub-agent execution policies from branch helpers to thread helpers.
- Update content upload/download clients to use thread-scoped routes.
- Move provider-specific stored config from `AdditionalProperties` to `ProviderOptions`.
- Update TUI extensions to produce transcript cells and register renderers.
- Adapt model selection integrations to `IAgentTuiModelCatalog`.

---

## Release Posture

This release should be read as the current `0.x` API surface rather than a compatibility-preserving patch. HPD-Agent remains pre-1.0, and changes like this are expected until `1.0.0`. The documentation should describe the thread model as the normal model going forward.
